### [LƯỢT CHAT 1] PHẢN HỒI CỦA AI VỀ BƯỚC 1: SO SÁNH KAFKA VÀ RABBITMQ

Chào bạn, dưới đây là bảng so sánh chi tiết và tư vấn thiết kế Message Broker cho hệ thống SpeedyCart của bạn:

#### Bảng so sánh Apache Kafka vs RabbitMQ:

| Tiêu chí | Apache Kafka | RabbitMQ |
| :--- | :--- | :--- |
| **Kiến trúc lõi** | Nhật ký phân tán (Distributed Commit Log). Dữ liệu được lưu trữ tuần tự trên đĩa cứng. | Hàng đợi tin nhắn truyền thống (Message Queue). Tin nhắn được xóa sau khi tiêu thụ thành công. |
| **Mô hình truyền tải** | **Pull-based:** Consumer chủ động kéo dữ liệu từ broker khi sẵn sàng. | **Push-based:** Broker chủ động đẩy tin nhắn đến Consumer đang kết nối. |
| **Hiệu năng (Throughput)** | **Cực kỳ cao:** Lên đến hàng triệu tin nhắn/giây nhờ cơ chế ghi tuần tự và Zero-copy. | **Trung bình-Cao:** Khoảng hàng chục nghìn tin nhắn/giây. |
| **Độ trễ (Latency)** | Thấp (Khoảng vài mili-giây). | Cực kỳ thấp (Gần như tức thời - micro-seconds). |
| **Khả năng mở rộng** | Rất dễ dàng thông qua việc phân chia Partitions trên nhiều node (Horizontal Scale). | Phức tạp hơn, giới hạn bởi dung lượng RAM của node chính chứa queue. |
| **Độ tin cậy & Lưu trữ** | Tuyệt vời. Tin nhắn được lưu trữ vĩnh viễn trên đĩa theo thời gian cấu hình (Retention Period). | Khá tốt. Tin nhắn có thể bền bỉ (persistent) nếu cấu hình ghi đĩa, nhưng sẽ bị xóa khi xử lý xong. |

#### Đề xuất lựa chọn cho SpeedyCart:
- **Nên chọn Apache Kafka** cho kịch bản này. Lý do:
  - Sự kiện `OrderCompletedEvent` cần được tiêu thụ bởi nhiều dịch vụ độc lập (Email, Loyalty, Warehouse). Kafka sử dụng cơ chế **Consumer Group**, mỗi dịch vụ là một group độc lập đọc từ cùng một Topic mà không ảnh hưởng lẫn nhau.
  - Khả năng lưu trữ (Retention) của Kafka cho phép các dịch vụ có thể "xem lại" (replay) hoặc tiêu thụ lại các tin nhắn cũ nếu dịch vụ đó gặp sự cố sập nguồn.

### [LƯỢT CHAT 2] PHẢN HỒI CỦA AI VỀ BƯỚC 2: PHÂN TÍCH KỊCH BẢN REDIS IDEMPOTENCY

Khi **Email Service bị sập nguồn trong 3 tiếng**:

#### 1. Tin nhắn trên Broker có bị mất không?
- **Với Apache Kafka:** Tin nhắn hoàn toàn **KHÔNG** bị mất. Tin nhắn được ghi chặt xuống đĩa của Kafka Broker và lưu trữ theo thời gian cấu hình (mặc định là 7 ngày). Khi Email Service sập, Kafka chỉ lưu lại vị trí đọc cuối cùng (Offset) của nhóm Email Consumer. Khi dịch vụ sống lại, nó sẽ tiếp tục kéo tin nhắn từ Offset cũ đó và xử lý bình thường.
- **Với RabbitMQ:** Tin nhắn **KHÔNG** bị mất **nếu và chỉ nếu**:
  1. Queue được cấu hình là `Durable` (bền bỉ).
  2. Tin nhắn gửi đi được đánh dấu thuộc tính `deliveryMode = 2` (Persistent).
  Khi đó, tin nhắn được lưu trữ tạm thời trên đĩa của RabbitMQ và xếp hàng chờ đợi cho đến khi Email Service online trở lại để nhận dữ liệu (Ack).

#### 2. Giải pháp chống gửi trùng Email (Idempotency) sử dụng Redis
Khi Email Service hoạt động lại, cơ chế retry tự động hoặc tiêu thụ hàng loạt tin nhắn tồn đọng có thể khiến một email đơn hàng bị gửi đi nhiều lần. Chúng ta sử dụng **Redis làm Distributed Idempotency Key Store**:
- Mỗi sự kiện `OrderCompletedEvent` bắt buộc phải có một định danh duy nhất: `eventId` (UUID) hoặc `orderId`.
- **Giải thuật xử lý nguyên tử (Atomic Check-and-Set) bằng Redis:**
  1. Khi nhận sự kiện, Consumer tạo khóa: `key = "idempotency:email:" + eventId`.
  2. Gọi lệnh `SET key "PROCESSING" EX 86400 NX` (Trong Spring Boot là `setIfAbsent` kèm TTL 24 giờ). Lệnh này chỉ ghi thành công nếu key chưa tồn tại.
  3. Nếu Redis trả về `false` (key đã tồn tại):
     - Nếu giá trị là `"PROCESSING"` hoặc `"COMPLETED"`: Bỏ qua tin nhắn này ngay lập tức vì nó đang được xử lý hoặc đã xử lý xong.
  4. Nếu Redis trả về `true` (ghi khóa thành công):
     - Tiến hành gửi email thông qua Mail Server.
     - Nếu gửi thành công: Cập nhật giá trị key thành `"COMPLETED"` (vẫn giữ nguyên TTL).
     - Nếu gửi thất bại: Xóa key này khỏi Redis (để luồng retry sau này hoặc tin nhắn gửi lại có cơ hội thực thi lại).

### [LƯỢT CHAT 3] PHẢN HỒI CỦA AI VỀ BƯỚC 3: MÃ NGUỒN KAFKA CONSUMER VÀ DLQ CONFIGURATION

Dưới đây là toàn bộ mã nguồn Java Spring Boot triển khai logic Kafka Consumer tích hợp Redis để chống trùng lặp và cấu hình cơ chế Retry/DLQ.

#### 1. File cấu hình Spring Boot `application.yml`
```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    consumer:
      group-id: email-service-group
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
    listener:
      missing-topics-fatal: false
```

#### 2. Class cấu hình Kafka Error Handler và DLQ: `KafkaConsumerConfig.java`
```java
package com.speedycart.config;

import lombok.extern.slf4j.Slf4j;
import org.apache.kafka.clients.consumer.ConsumerRecord;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.KafkaTemplate;
import org.springframework.kafka.listener.CommonErrorHandler;
import org.springframework.kafka.listener.DeadLetterPublishingRecoverer;
import org.springframework.kafka.listener.DefaultErrorHandler;
import org.springframework.util.backoff.FixedBackOff;

@Slf4j
@Configuration
public class KafkaConsumerConfig {

    /**
     * Cấu hình Error Handler: Thử lại 3 lần, giãn cách 2 giây.
     * Nếu sau 3 lần vẫn lỗi, đẩy tin nhắn lỗi sang Topic DLQ: [tên_topic_gốc].DLQ
     */
    @Bean
    public CommonErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
        // Cấu hình hồi phục bằng cách gửi sang DLQ
        DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template,
                (consumerRecord, exception) -> {
                    log.error("All 3 retries failed for record key: {}. Routing to DLQ: {}.DLQ",
                            consumerRecord.key(), consumerRecord.topic());
                    return new org.apache.kafka.common.TopicPartition(
                            consumerRecord.topic() + ".DLQ", 
                            consumerRecord.partition()
                    );
                });

        // Retry 3 lần, mỗi lần cách nhau 2 giây
        FixedBackOff fixedBackOff = new FixedBackOff(2000L, 3);
        
        DefaultErrorHandler errorHandler = new DefaultErrorHandler(recoverer, fixedBackOff);
        // Chỉ cấu hình retry cho các lỗi liên quan đến Mail Server, tránh retry vô hạn cho lỗi code
        errorHandler.addRetryableExceptions(com.speedycart.exception.MailServerException.class);
        
        return errorHandler;
    }
}
```

#### 4. Class Kafka Consumer tích hợp Redis `OrderEventListener.java`
```java
package com.speedycart.listener;

import com.speedycart.event.OrderCompletedEvent;
import com.speedycart.exception.MailServerException;
import com.speedycart.service.EmailService;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.kafka.annotation.KafkaListener;
import org.springframework.stereotype.Component;

import java.time.Duration;

@Slf4j
@Component
@RequiredArgsConstructor
public class OrderEventListener {

    private final RedisTemplate<String, Object> redisTemplate;
    private final EmailService emailService;

    private static final String IDEMPOTENCY_KEY_PREFIX = "idempotency:email:";
    private static final Duration KEY_TTL = Duration.ofHours(24);

    @KafkaListener(topics = "order-completed", groupId = "email-service-group")
    public void handleOrderCompletedEvent(OrderCompletedEvent event) {
        String eventId = event.getEventId();
        String redisKey = IDEMPOTENCY_KEY_PREFIX + eventId;

        log.info("Received OrderCompletedEvent. Event ID: {}, Order ID: {}", eventId, event.getOrderId());

        // Bước CoT 1: Thực hiện ghi nhận trạng thái PROCESSING nguyên tử kèm TTL 24 tiếng
        Boolean isAbsent = redisTemplate.opsForValue().setIfAbsent(redisKey, "PROCESSING", KEY_TTL);

        if (Boolean.FALSE.equals(isAbsent)) {
            // Khóa đã tồn tại, kiểm tra xem trạng thái hiện tại là gì
            Object currentStatus = redisTemplate.opsForValue().get(redisKey);
            log.warn("Duplicate message detected for Event ID: {}. Current state in Redis: {}. Skipping execution.",
                    eventId, currentStatus);
            return;
        }

        try {
            // Bước CoT 2: Gọi dịch vụ gửi email
            log.info("Processing email delivery for Order ID: {} to {}", event.getOrderId(), event.getCustomerEmail());
            emailService.sendOrderConfirmationEmail(event.getCustomerEmail(), event.getOrderId());

            // Bước CoT 3: Gửi thành công -> Cập nhật trạng thái sang COMPLETED
            redisTemplate.opsForValue().set(redisKey, "COMPLETED", KEY_TTL);
            log.info("Email sent successfully and idempotency key updated to COMPLETED for Event ID: {}", eventId);

        } catch (MailServerException e) {
            // Bước CoT 4: Gặp lỗi Mail Server -> Xóa key trong Redis để có thể chạy lại khi Kafka Retry kích hoạt
            redisTemplate.delete(redisKey);
            log.warn("MailServerException occurred. Deleted Redis key: {} to allow retry. Error: {}", redisKey, e.getMessage());
            // Ném ngoại lệ để kích hoạt cơ chế Kafka Retry / DLQ
            throw e;
        } catch (Exception e) {
            // Lỗi lập trình hoặc lỗi nghiệp vụ không thể phục hồi -> Không xóa key để tránh loop lỗi vô hạn
            log.error("Fatal non-retryable error occurred for Event ID: {}. Error: {}", eventId, e.getMessage(), e);
            redisTemplate.opsForValue().set(redisKey, "FAILED", KEY_TTL);
            // Không ném lại exception để tránh kích hoạt retry vô ích
        }
    }
}
```

#### 5. Class giả lập Dịch vụ Email `EmailService.java`
```java
package com.speedycart.service;

import com.speedycart.exception.MailServerException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

@Slf4j
@Service
public class EmailService {

    private int attempt = 0;

    public void sendOrderConfirmationEmail(String email, String orderId) {
        // Giả lập thỉnh thoảng lỗi mạng khi gọi SMTP Server
        attempt++;
        if (attempt % 3 != 0) { 
            log.warn("SMTP Connection timeout. Failed to connect to Mail Server.");
            throw new MailServerException("SMTP Mail server is temporarily down. Attempt: " + attempt);
        }
        
        log.info("Email confirmation sent successfully to {} for order {}", email, orderId);
        attempt = 0; // Reset
    }
}
```
